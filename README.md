# MyTCP

一个基于 **Java** 实现的可靠传输协议实验项目，实现了从 **RDT1.0 到 TCP Reno** 的完整协议演进过程。项目在不可靠信道环境下逐步实现差错检测、超时重传、滑动窗口以及 TCP 拥塞控制等机制，用于理解可靠传输协议和 TCP 拥塞控制算法的设计原理。 TCP Reno

------

# 项目简介

在真实网络环境中，数据传输可能会出现以下问题：

- 数据包 **丢失**
- 数据包 **损坏（bit error）**
- 数据包 **延迟**
- 数据包 **乱序到达**

本项目在模拟的不可靠信道上逐步实现可靠传输协议，从最基础的 **RDT1.0** 开始，逐步扩展到 **TCP Tahoe** 和 **TCP Reno**，完整复现可靠传输协议的发展过程。

协议演进流程如下：

```
RDT1.0
   ↓
RDT2.x（差错检测）
   ↓
RDT3.0（超时重传）
   ↓
Selective Repeat（滑动窗口）
   ↓
TCP（可靠传输）
   ↓
TCP Tahoe（拥塞控制）
   ↓
TCP Reno（快恢复）
```

# 已实现功能

## 1. RDT1.0

- 假设信道完全可靠
- 不存在丢包、错误或延迟
- 实现基础的 **发送—接收—确认（ACK）机制**

------

## 2. RDT2.x

在 RDT1.0 的基础上，引入 **差错检测机制**：

- 使用 **CRC32** 计算校验和
- 接收端检测数据是否损坏
- 采用 **ACK机制进行错误恢复**
- 使用 **重复ACK代替NAK**

------

## 3. RDT3.0

在 RDT2.x 基础上进一步支持：

- **数据包丢失**
- **确认包丢失**
- **网络延迟**

新增机制：

- **定时器（Timer）**
- **超时重传（Timeout Retransmission）**

当发送端在规定时间内未收到 ACK 时，会自动重传对应数据包。

------

## 4. Selective Repeat（SR）

为了解决 RDT3.0 停等协议效率低的问题，引入 **滑动窗口协议**：

主要特性：

- 发送端支持 **流水线发送**
- 接收端支持 **乱序缓存**
- **按分组确认（Per-packet ACK）**
- 仅重传 **出错或丢失的分组**

这大大提高了信道利用率。

------

## 5. TCP（基础版本）

实现一个 **不包含拥塞控制的 TCP 可靠传输协议**。

主要特点：

发送端：

- 使用 **GBN 风格滑动窗口**
- 窗口大小为 **16**
- 使用 **统一计时器进行超时重传**

接收端：

- 使用 **SR方式缓存乱序数据**
- 使用 **累计确认（Cumulative ACK）**

------

## 6. TCP Tahoe

在 TCP 基础上实现 **拥塞控制机制**。

实现的算法包括：

- **Slow Start（慢启动）**
- **Congestion Avoidance（拥塞避免）**
- **Fast Retransmit（快重传）**

关键变量：

```
cwnd       拥塞窗口
ssthresh   慢启动阈值
```

当检测到拥塞时：

```
ssthresh = cwnd / 2
cwnd = 1
```

然后重新进入慢启动阶段。

------

## 7. TCP Reno

TCP Reno 在 Tahoe 基础上加入 **Fast Recovery（快恢复）机制**。

当收到 **3 个重复 ACK** 时：

```
ssthresh = cwnd / 2
cwnd = ssthresh + 3
```

这样可以避免重新进入慢启动阶段，使网络吞吐量更高。

# 实验结果

在启用以下不可靠信道条件下进行测试：

- 数据包损坏
- 数据包丢失
- ACK延迟

实验结果表明：

- 数据能够 **可靠传输**
- 接收端数据 **无重复、无丢失**
- 数据 **按序交付**
- TCP Reno 在平均情况下 **性能优于 Tahoe**
